# Content Strategy · Guizang Magazine

怎么写，不只是怎么排。Guizang 风格的文档不是“有样式的公文”，而是**被编辑过的内容**。

---

## 核心原则

### 1. 数据优于形容词

- 少写“显著”“优秀”“全面”
- 多写数字、时间、规模、变化幅度

### 2. headline 要像标题，不像栏目名

主标题要有抓力。能让人停下来读。

- 不要：`关于 XXX 项目的阶段性进展汇报`
- 更好：`从试验到可用：XXX 已经跨过了哪条线`

### 3. kicker 负责“钩子”，chrome 负责“导航”

两者不能重复。

- `chrome`：稳定栏目，如 `Act II · Workflow`
- `kicker`：本页独有的短句，如 `先看数据，再谈判断`

### 4. 正文是解释，不是重复标题

标题负责抛结论，正文负责补证据、补过程、补边界。

### 5. 句子要有节奏

Guizang 风格适合：

- 短句
- 断开的层级
- 一眼能扫到的信息块

不适合：

- 一整页糊成一坨的长段落
- 套话型总结

## 文档类型策略

### One-Pager

目标：30 秒抓住核心。

推荐顺序：

1. headline
2. 3-4 个 metrics
3. 一段 lead
4. 关键判断 / 证据 / 下一步

要求：

- headline 自带判断
- metrics 尽量短
- 每个 section 标题读起来像摘要

### Long Doc

目标：读者可以持续读下去，而不是只停在封面。

要求：

- 每章开头要先讲“本章结论”
- 每 2-4 页要有一次节奏变化
- 数据、引语、图示穿插，不能全是正文

### Letter

目标：正式、清楚、有温度，但不空泛。

推荐结构：

1. sender meta
2. subject line
3. salutation
4. 关系 / 背景
5. 证据 / 判断
6. 明确建议或结论

### Resume

目标：像一份有编辑意识的职业档案。

要求：

- 名字和定位是封面信号
- metrics 反映持续性、规模和结果
- 项目经历写成：角色 -> 动作 -> 结果

### Portfolio

目标：像一本项目专刊，不像截图堆砌。

要求：

- 先给项目标题和一句话定义
- 图片服务于叙事，不要漫无目的铺图
- 每个项目必须点出挑战和结果

### Slides

目标：延续 Guizang deck 的叙事弧。

推荐节奏：

- cover
- numbers
- explanation
- transition
- quote / tension
- resolution

## 统一格式规则

### 数字

- `5,000+` 而不是 `5000+`
- `90%` 而不是 `90 %`
- 能模糊就别假精确

### 标签

- 短、硬、可扫读
- 最好 1-3 个词

### 引语

- 只放真正有力量的话
- 不要用引号包装普通结论

### 结论句

结论要像判断，不像总结陈词。

- 不要：`综上所述，我们取得了阶段性成果`
- 更好：`真正的变化，不是效率更高，而是一个人开始能完成过去十个人的工作`

---

## 交付前自查

- 有没有废话型标题？
- 有没有页眉和 kicker 重复？
- 有没有一页里超过 2 处高亮？
- 有没有连续几页节奏完全一样？
- 有没有数字能再具体一点？
- 有没有段落其实应该拆成卡片或列表？

一句话：**Guizang 风格最怕的不是“不够花”，而是“没有被编辑过”。**
